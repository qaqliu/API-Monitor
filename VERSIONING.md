# Versioning Policy

API Monitor uses SemVer-compatible release versions: `x.y.z`.

Keep official release versions compatible with `package.json`, GitHub Releases, and `electron-updater`. Do not use four-part numeric versions such as `x.y.z.n` for official releases.

## Version Meaning

- `x++` major: architecture-level refactors, incompatible store/data-flow changes, packaging/update-flow rewrites, or changes that require explicit migration.
- `y++` minor: add a new model/provider type, add a new monitoring category, or introduce a significant new feature while keeping existing configuration compatible.
- `z++` patch: modify existing provider/model behavior, fix bugs, adjust UI copy/styles, improve docs, or make smaller maintenance changes.

## Release Rules

- Every published release must update `package.json` `version`.
- Git tags must match the package version exactly: `vX.Y.Z`.
- Automatic updates are published from GitHub Releases generated by `v*` tags.
- Smaller-than-patch maintenance still increments `z`; describe it in release notes as a hotfix, docs update, config update, or maintenance update.

## Examples

- `1.0.1`: fix the existing Codex provider display or README links.
- `1.1.0`: add a new provider/model type.
- `2.0.0`: rewrite the provider architecture or change persisted configuration in a way that needs migration.
